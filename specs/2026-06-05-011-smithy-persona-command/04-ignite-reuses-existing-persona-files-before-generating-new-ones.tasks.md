## Slice 1: Discover and Match Durable Personas in Ignite

**Goal**: Amend ignite Sub-phase 3b so it discovers existing `.persona.md` files through the canonical persona convention before any cold persona drafting occurs.

**Justification**: This slice creates the reuse decision point without changing the final RFC prose generation path. It is independently valuable because an ignite run can identify which durable personas are available, which requested personas they cover, and which personas still need cold drafting.

**Addresses**: FR-011; AS 4.1

### Tasks

- [ ] **Add pre-draft persona discovery**

  Update `src/templates/agent-skills/commands/smithy.ignite.prompt` so Sub-phase 3b reads the README-defined persona convention and checks the resolved persona directory before dispatching `smithy-prose` for Personas. Keep discovery scoped to the active `{{artifactsRoot}}` so in-repo and external-artifacts modes do not cross-contaminate persona stores.

  _Acceptance criteria:_
  - Sub-phase 3b discovers existing `.persona.md` files before any Personas cold-draft dispatch.
  - Discovery is tied to the canonical persona convention rather than an independently restated schema.
  - If no persona files exist, ignite follows the existing cold-draft path.
  - Discovery does not change RFC-mode behavior for `smithy.persona`.

- [ ] **Match needed personas by canonical slug identity**

  Define the Sub-phase 3b reuse match as a deterministic filename-slug comparison: derive kebab-case slugs from personas named or clearly described during Phase 2 clarification, and treat matching `<slug>.persona.md` files as covering those personas. Avoid fuzzy matching, interactive selection, or a new registry.

  _Acceptance criteria:_
  - A clarification-surfaced persona whose derived slug has a matching file is marked covered.
  - A clarification-surfaced persona without a matching file remains a gap for cold drafting.
  - Matching relies on filename slug identity, consistent with the persona convention.
  - The prompt does not introduce interactive matching that would break ignite's one-shot flow.

**PR Outcome**: Ignite Sub-phase 3b can determine reusable persona coverage before generation while preserving the existing cold path when no files match.

## Slice 2: Project Reused Personas and Draft Only Gaps

**Goal**: Use the Sub-phase 3b coverage decision to build the RFC `## Personas` section from durable persona files for covered personas and cold-draft only uncovered gaps.

**Justification**: This slice delivers the user-visible US4 behavior: reused personas appear in the RFC as RFC-specific benefit projections, and new prose is generated only where durable coverage is absent.

**Addresses**: FR-012, FR-013; AS 4.2, AS 4.3

### Tasks

- [ ] **Project covered personas into the RFC**

  Update Sub-phase 3b instructions so each covered persona file is read and projected into the RFC `## Personas` section as how this RFC benefits that durable persona. The projection should preserve the durable persona's role, context, and friction while tailoring the final benefit language to the current RFC.

  _Acceptance criteria:_
  - Covered personas are drafted from their existing `.persona.md` files rather than invented cold.
  - The RFC `## Personas` output describes how this RFC benefits each covered persona.
  - Projection keeps durable persona content as the source context while allowing RFC-specific benefit framing.
  - The resulting section still passes ignite's existing non-empty Personas verification.

- [ ] **Cold-draft only uncovered persona gaps**

  Amend the `smithy-prose` dispatch in Sub-phase 3b so cold drafting is limited to personas not covered by existing files, then combine any cold-drafted gap content with the file-sourced projections into one valid `## Personas` section.

  _Acceptance criteria:_
  - Personas with matching files are not re-generated by the cold Personas dispatch.
  - Personas without matching files are still generated through the existing `smithy-prose` Personas path.
  - The final RFC contains one `## Personas` section that includes both reused and newly drafted personas when both are present.
  - If all needed personas are covered by files, Sub-phase 3b does not perform an unnecessary cold Personas dispatch.

- [ ] **Protect reuse behavior with template tests**

  Update template coverage around `smithy.ignite.prompt` so the reuse-before-generate behavior is protected without regenerating deployed snapshots. Focus the checks on discovery-before-dispatch, slug-based coverage, projection from files, and gap-only cold drafting.

  _Acceptance criteria:_
  - Tests or template checks cover discovery before cold drafting.
  - Tests or template checks cover slug-based covered-vs-gap behavior.
  - Tests or template checks cover file projection and gap-only cold drafting.
  - No `.claude/`, `.gemini/`, `.agents/`, `.codex/`, or manifest snapshot files are regenerated.

**PR Outcome**: With durable persona files present, ignite reuses matching personas in the RFC and generates fresh persona prose only for uncovered gaps.
