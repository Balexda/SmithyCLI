## Slice 1: Plant the render-eval RFC fixture and document it

**Goal**: A scenario-isolated directory `evals/fixture/rfcs/render-eval/` exists containing a minimal-but-structurally-valid RFC plant (`render-eval.rfc.md`) with at least one `### Milestone 1:` heading (the AS 4.1 precondition), and `evals/fixture/README.md` carries a `## Planted Parent Artifacts` row referencing the new plant directory — creating the section if it does not yet exist.

**Justification**: The RFC plant is pure static content that Slice 2's YAML references by exact path in its `prompt` field. Landing it first settles the slug, lets reviewers see the plant content in isolation, ensures Slice 2's `prompt` resolves on its first authored draft, and matches the US1/US2/US3 precedent of co-locating plant authoring with its README documentation row in a single slice. The slice has no code dependency on US2's runner git-init — the plant itself is pure static content and merge-safe regardless of sibling ordering. Co-locating the README row with the plant keeps the "this is a deliberate fixture, do not delete" signal next to the file it protects.

**Addresses**: FR-004, FR-005, FR-013, FR-014; AS 4.1 (planted-fixture precondition: RFC contains at least one `### Milestone N:` heading); FR-012 (existing fixture untouched).

## Slice 2: Author the render-from-rfc YAML scenario

**Goal**: `evals/cases/render-from-rfc.yaml` exists, is auto-discovered by `loadScenarios`, and `npm run eval -- --case render-from-rfc` reports PASS against the Slice 1 plant once US2 Slice 1's runner git-init is present, with sub-agent evidence covering smithy-scout, smithy-clarify, and smithy-plan-review per the spec's Sub-Agent Evidence Matrix.

**Justification**: The YAML scenario is the user-facing deliverable that closes AS 4.1 and AS 4.2. It depends on Slice 1 because its `prompt` field references the Slice 1 plant by exact path. Slice 2 also has a **hard cross-story dependency** on US2 Slice 1 (runner git-init): render's Phase 4 dispatches `git checkout -b` inside the temp fixture copy, which fails before any one-shot output is captured if the temp copy is not a git repository. Slice 2 MUST NOT merge until US2 Slice 1 has merged — see the Cross-Story Dependencies table. AS 4.1's "references to the rendered features-map path" cannot be asserted against the on-disk `.features.md` (per inherited SD-010, the runner only validates `extracted_text` from stream-json); the scenario uses the terminal one-shot snippet's `**Artifacts produced**:` bullet (which lists the rendered `.features.md` basename) as the load-bearing proxy invariant, anchored alongside `**Spec folder**:` to catch both successful Phase 4 completion and the produced features-map filename surface. Empirical timeout calibration, the PR success/failure regex alternation, and the `forbidden_patterns` set are YAML-field-level concerns that belong here. The smithy-plan deviation flagged by competing-lens analysis is carried as SD-013 below so the implementer can promote `sub_agent_evidence` from three to four entries if empirical observation during calibration confirms the dispatch.

**Addresses**: FR-001, FR-002, FR-003, FR-005, FR-009, FR-010, FR-011, FR-012, FR-014; AS 4.1 (one-shot-snippet headings + `**Artifacts produced**:` bullet listing the `.features.md`), AS 4.2 (sub-agent evidence for smithy-scout, smithy-clarify, smithy-plan-review per spec matrix), AS 4.1/4.2 source-fixture-checksum invariant (via the runner's existing `hashDirectory` guards); SC-001, SC-002, SC-005, SC-006.

Field-level constraints (the regex-alternation strategy for SD-008, the matrix-faithful three-agent `sub_agent_evidence` plus the SD-013 deviation note, the empirical timeout calibration per SD-014, and the SD-015 plan-review-marker capture) are enumerated below.

  _Acceptance criteria:_
  - `name: render-from-rfc`; `skill: /smithy.render`; `prompt` is the exact repo-relative path to the Slice 1 RFC plant
  - `required_headings` contains the four one-shot-snippet section headings stamped by render's Phase 4 (AS 4.1)
  - `required_patterns` contains the `\*\*Spec folder\*\*` marker
  - `required_patterns` contains a pattern that anchors to the `\*\*Artifacts produced\*\*` bullet plus a `.features.md` substring match (AS 4.1's features-map-path reference)

  - `sub_agent_evidence` has three entries covering smithy-scout, smithy-clarify, and smithy-plan-review, each with a template-stable pattern per contracts §3 marker conventions (AS 4.2; spec matrix)
  - The plan-review entry uses an empirically captured template-stable marker pinned in an inline YAML comment (resolves SD-015)
  - An inline YAML comment on the `sub_agent_evidence` block documents the matrix-vs-runtime gap for smithy-plan and cross-references SD-013
  - `forbidden_patterns` ⊇ the three strike-convention entries (FR-011)
  - `timeout` is set to a positive empirically calibrated number of seconds (not the 120s default), with an inline comment recording the observed wall-clock (FR-010, inherited SD-006, re-tracked as SD-014)
  - YAML loads without warning via `loadScenarios`; `npm run eval -- --case render-from-rfc` selects only this scenario and reports PASS once US2 Slice 1 is in place (SC-001, SC-002)
  - Strike, scout, and any landed sibling scenarios continue to PASS unchanged after this scenario is added (FR-012, SC-003)
  - `npm run test:evals` (offline unit tests) continues to pass with the new YAML file present (SC-006)

**PR Outcome**: `npm run eval -- --case render-from-rfc` exercises `/smithy.render` against the planted RFC artifact and reports PASS, with sub-agent evidence for the three matrix-mandated dispatches and `**Artifacts produced**:` confirming AS 4.1's features-map-path reference. The scenario passes whether `gh` PR creation succeeds or falls through to the PR-creation-failure branch. Inherited debt SD-006, SD-008, SD-010, SD-011 transition toward `resolved` for the render variant once the YAML scenario lands and is calibrated.
