**Questions & Answers**:
- Q: Beyond the three personas in the PRD, are token-savings PR reviewers a distinct stakeholder for this RFC? → A: Yes. F3.4 introduces a PR-description token-delta protocol that maintainers consume read-only without re-running evals locally; their workflow differs from contributors who run evals before opening a PR. Add them as a fourth persona.
- Q: How should the 40% per-slice token-reduction target be framed in the RFC? → A: Carry it forward as a numeric target with an inline SD-008 calibration footnote, not as "TBD". The estimate drives prioritization across M2/M3 and is recoverable: post-M1 baselines may adjust it upward or downward, but the RFC needs a concrete number to gate Success Metrics.
- Q: How tight is the cross-RFC dependency on expand-evals US1–US3? → A: Hard prerequisite for M3 F3.2 model-line edits, soft prerequisite for M3 F3.2 template drafting. M1 measurement work proceeds independently. The reconciled plan's gating rule (F3.2 gated on M1 only at the milestone level) is refined here at the slice level: the model-downgrade slices wait on expand-evals merge to master.
- Q: Is the `.claude/` snapshot policy an RFC-level constraint or just an implementation detail? → A: RFC-level. Every milestone touches `src/templates/` extensively, and a snapshot-refresh chore (F3.5) is explicitly part of M3. Codifying the constraint in the RFC body — not just in the debt table — prevents accidental snapshot regeneration in feature PRs.
- Q: Why are M1 F1.4 (fix) and F1.5 (forge) eval-coverage features in M1 rather than M2? → A: Invocation-frequency hierarchy `fix > forge > cut > rare planners` means measurement on the highest-frequency commands is the highest-leverage M1 work. Without forge/fix coverage in M1, the M2 cost-reduction features (F2.3, F2.4, F2.5) have no measurement substrate to validate against.
- Q: How are M2 and M3 template-edit conflicts prevented if they run in parallel? → A: Via a normative touched-files single-owner matrix in the M0 governance section. Each prompt file has exactly one owning milestone for the program duration. SD-007 records the residual risk; the matrix elevates it from commentary to enforcement.
- Q: What happens to per-sub-agent attribution (F1.3b) if SD-001 resolves negative? → A: F1.3b is descoped to a post-M1 follow-up; M1 completes on F1.3a (per-case totals + baselines) plus F1.1/F1.2/F1.4/F1.5/F1.6. The reconciled plan's two-feature split for F1.3 already encodes this contingency.
- Q: Should `smithy.implement`, `smithy.cut` Phase 2.8 fan-out, `smithy.survey` model assignment, or the Anthropic Batches API integration be in scope? → A: No to all four. `smithy.implement` is explicitly deferred per SD-004 (revisited post-M2.1). `smithy.cut` Phase 2.8 fan-out is deferred to a future RFC per PRD Open Questions. `smithy.survey` is rejected as wrong shape (it's a one-shot landscape survey, not a high-frequency command). Batches API is orthogonal — it reduces per-message price, not per-message token count, which is this program's lever.
