# RFC: Smithy Token-Savings Program

**Created**: 2026-05-12  |  **Status**: Draft

## Summary

This RFC proposes a three-milestone token-savings program for Smithy CLI — a measurement foundation (M1) followed by architectural cost reductions (M2) running in parallel with low-risk early wins (M3) — targeted at the high-frequency commands `smithy.fix`, `smithy.forge`, and `smithy.cut`. Smithy's output quality is already strong; what is rising unsustainably is per-run cost, and without per-case and per-sub-agent token reporting in the eval framework every proposed optimization is a guess.

## Motivation / Problem Statement

A smithy power user runs `smithy.fix` on basically every PR, `smithy.forge` on the second-most-common workflow, and `smithy.cut` whenever a story needs slicing. These three commands dominate day-to-day invocations and their outputs are consistently good — the quality story is not the problem. The cost story is. Per-run token spend has climbed steadily as smithy has grown more sophisticated, and the trajectory is unsustainable relative to how often these commands now run across a user's PR flow. The pain is not occasional sticker shock on a rare expensive command; it is the same handful of commands being invoked many times a day, each one costing more than it did a release ago.

The cost is concentrated in three identifiable layers, and each layer has a different correction. The first is sub-agent model selection: ten of thirteen sub-agents run on Opus, and several of them are categorical scan-and-triage roles producing structured findings against fixed checklists — the workload Sonnet handles well at a fraction of the cost. The second is artifact re-read cascades: `smithy.forge` dispatches a fresh Opus sub-agent per task, and each one re-loads the same spec, data-model, and contracts from a cold context to look up acceptance scenarios for a single task. A ten-task slice can spend up to 300K tokens on re-reads before any reasoning happens, and a typical cut-plus-forge workflow accumulates roughly 264K tokens of spec re-reads across its phases. The third is build-tool output handling: there is no prescribed protocol for terse-mode flags, log-file capture, or tail bounds when the system runs `npm test`, `gradle`, `pytest`, or fetches a CI log via `gh run view`. CI logs alone routinely run 50K–500K tokens of mostly-noise, and today they land in agent context unfiltered. The Docker skill already demonstrates the right bounding pattern, but it has not been generalized.

What blocks confident action on any of the three layers is measurement. The eval framework today reports structural pass/fail but not token usage, and it only exercises `strike` and `scout` — neither of which is heavily used in practice — on a JavaScript-only fixture. The two commands that drive most of the cost (`forge` and `fix`) have zero automated coverage, and the in-flight expand-evals work deliberately excludes them. Without per-case and per-sub-agent token reporting, committed baselines, forge-shaped scenarios, and a multi-language fixture, every optimization in this space ships on faith. The goal of this RFC is to close that measurement gap first, then use it to land the architectural and quick-win cost reductions with a defensible token delta on every PR — so smithy's quality remains intact while per-invocation cost on the high-frequency commands drops materially.

## Goals

- Every `npm run eval` report attributes input and output token counts per scenario and per dispatched sub-agent, so that any future template change can be evaluated for cost impact alongside structural pass/fail. Per-sub-agent attribution is contingent on SD-001 evidence captured during M1; if the Claude CLI stream-json attributes sub-agent usage only to the parent transcript, this goal narrows to per-case totals plus committed baselines and per-sub-agent attribution becomes a post-M1 follow-up.
- The eval framework exercises both `smithy.forge` and `smithy.fix` end-to-end, each with at least one committed scenario covering a high-cost path, so that the two highest-frequency cost-driving commands have automated measurement substrate before any M2 cost-reduction feature lands. Forge coverage runs against a JVM fixture in addition to the existing JavaScript fixture.
- A typical `smithy.forge` slice on the JavaScript fixture costs at least 40% fewer tokens than the M1 baseline (target; calibrated post-M1 baseline per SD-008), with no regression in structural-eval pass rate and no human-reviewed quality regression on at least two sampled outputs per affected sub-agent.
- Quality is non-negotiable across every milestone: no M2 or M3 slice lands that regresses structural-eval pass rate on the committed scenario set or shows a human-reviewed quality regression on the sampled outputs. Sub-agent model downgrades and orchestration changes ship only after their quality net (landing-soon expand-evals scenarios per SD-003, captured forge runs per SD-006) clears.
- M1 is the program-wide gate; M2 and M3 are constructed in parallel after M1 completes. M1 owns the measurement substrate (token-usage reporting in `EvalReport`, the JVM multi-language fixture, the `smithy.forge` and `smithy.fix` eval scenarios, committed baselines). M2 owns the architectural reductions (per-task spec re-reads in `smithy.implement.prompt`, `smithy.forge.prompt` Phase 6 orchestration for inline maid, trivial-slice review-skip heuristic). M3 owns the early wins (build-output protocol, four sub-agent model downgrades, `smithy.fix.prompt` CI-log handling, PR-description protocol, snapshot-refresh chore). The touched-files single-owner matrix in the Cross-Cutting Governance section keeps the two parallel milestones conflict-free.
- Every command that runs build, test, or CI tooling references the new build-output protocol snippet, so that raw verbose `npm test`, `gradle`, `pytest`, and `gh run view` output no longer lands unbounded in any agent's context window, and every smithy template PR after M3 carries a token-delta line in its PR description that maintainers can read without re-running evals locally.

## Out of Scope

- `smithy.implement` Opus-to-Sonnet model migration. Deferred per SD-004 and revisited only after M1 measurement plus F2.3 spec-reread reductions land; the post-F2.3 data is the decision criterion, not this RFC.
- `smithy.cut` Phase 2.8 competing-lens fan-out reduction (three parallel slice agents plus reconcile). Deferred to a possible follow-up RFC per the PRD Open Questions; M1 baselines will surface whether this fan-out is a larger cost share than currently estimated, but no slice in this program touches it.
- `smithy.survey` model assignment. Rejected as wrong-shape for this program: low-volume, web-research-driven synthesis rather than scan-and-triage, and its invocation frequency does not warrant a slice under the `fix > forge > cut > rare planners` hierarchy.
- Anthropic Messages Batches API integration. Orthogonal lever — reduces per-message price, not per-message token count, which is this program's only lever. Remains available as a future feature and is not precluded by anything in this RFC.
- Prompt-caching reliance as a cost-reduction mechanism. Cache lifetime is five minutes within a single conversation; sub-agents are spawned with fresh contexts and cannot inherit a parent cache, so the dominant smithy cost shape (per-task fresh sub-agents re-reading specs) is exactly the workload prompt caching does not help.
- External LLM observability platforms (LangSmith and equivalents). Adds an outside dependency and a separate UI for a less precise signal than instrumenting smithy's existing eval runner, which already parses the stream-json output that carries `usage.input_tokens` and `usage.output_tokens`.
- Cargo, Go, and Python fixtures beyond the JVM addition. The PRD assumption that a minimal JVM workload under `evals/fixture/jvm/` is sufficient to validate the gradle clauses of the build-output protocol stands; additional language coverage is nice-to-have and not blocking, and no slice in this program builds it.
- Modifying or extending the in-flight `specs/2026-05-03-005-expand-evals-coverage-planning-and-audit/` spec. That spec owns the audit, mark, cut, render, ignite, and spark scenarios, the YAML loader, and the baseline library; this RFC consumes them as dependencies via the Cross-Cutting Governance section and does not edit them.
- Deprecation of existing prompt patterns, API renames, migration tooling, or any user-facing CLI surface changes. The program is template edits and eval-framework instrumentation only; there is no public API change requiring a migration story.
- Regenerating `.claude/` or `.smithy/smithy-manifest.json` inside per-feature or per-bugfix PRs in this program. F3.5 is the dedicated snapshot-refresh chore at program end, with equivalent dedicated chore PRs (not features) at the close of M1 and M2. Per-feature edits in M1, M2, and M3 land under `src/templates/`, `evals/`, or `CONTRIBUTING.md` as the touched-files matrix specifies, but the committed `.claude/` snapshot is only ever updated by the dedicated chore PRs — per the Cross-Cutting Governance section and CLAUDE.md's "Source vs. Deployed Artifacts" section.

## Personas

- **Smithy power users** — Run `smithy.fix`, `smithy.forge`, and `smithy.cut` many times a day across their PR flow and currently watch per-run token spend climb each release with no measurement to point at the worst offender; this RFC gives them per-case and per-sub-agent token reporting plus the architectural and quick-win reductions that bring the dominant high-frequency commands back into a sustainable cost envelope.
- **Smithy contributors** — Propose prompt and sub-agent changes today with no defensible way to show whether a diff is net-positive on cost as well as quality, so optimization PRs ship on intuition; this RFC gives them committed baselines and a token-delta they can compute locally before opening a PR.
- **Future smithy adopters** — Weigh smithy against onboarding decisions that partly depend on predictable per-PR cost, and the current upward trajectory is a barrier to that adoption; this RFC stabilizes per-invocation cost on the commands that dominate real-world use and makes the cost story legible.
- **Smithy maintainers reviewing token-savings PRs** — Assess incoming template changes from a PR description alone, without re-running evals locally, and today have no consistent signal on cost impact; this RFC's F3.4 PR-description token-delta protocol (shipped via the M3 CONTRIBUTING.md update) gives them a read-only, at-a-glance verdict on whether a change is net-positive on tokens.

## Proposal

This RFC delivers a measurement-first, three-milestone program that brings the per-invocation cost of Smithy's three highest-frequency commands — `smithy.fix`, `smithy.forge`, and `smithy.cut`, in that priority order — back into a sustainable envelope without touching their output quality. M1 establishes the measurement substrate the program runs on: token-usage reporting in every eval report, committed baselines so any future template change can be assessed for cost delta as well as structural pass/fail, and end-to-end eval coverage for the two highest-frequency cost-driving commands (`smithy.forge` and `smithy.fix`) that currently have none. Crucially, the forge eval and the multi-language (JVM) fixture both sit in M1 rather than M2, because they are measurement infrastructure — neither one reduces cost on its own; both are preconditions for honestly evaluating any M2 reduction.

Once M1 closes, M2 and M3 construct in parallel against a single-owner touched-files matrix that keeps their template edits conflict-free. M2 owns the architectural reductions that need M1's measurement to ship safely — reducing the per-task spec re-read cascade in `smithy.forge`, inlining a low-cost helper sub-agent into the forge primary, and skipping the review sub-agent on trivial slices via a calibrated heuristic. M3 owns the low-risk early wins that show up against existing and landing-soon planning-command scenarios — a generalized build-output protocol that bounds verbose `npm test` / `gradle` / `pytest` / `gh run view` output across every command, four sub-agent model downgrades from Opus to Sonnet for scan-and-triage roles, smarter CI-log handling in `smithy.fix`, and a PR-description token-delta protocol so maintainers can read a cost verdict without re-running evals locally. The user-observable outcome is that quality remains intact while per-invocation cost on the dominant commands drops materially, and every Smithy template PR after M3 carries a defensible token delta in its description.

## Design Considerations

- **Sub-agent dispatch is the dominant cost shape, and prompt caching cannot rescue it.** A Smithy sub-agent is a fresh process with a cold context window: it re-reads every spec, data-model, and contracts file it needs, because it inherits nothing from the parent. That architecture is load-bearing for parallelism and isolation, but it also means each per-task forge dispatch pays the full cold-context re-read cost, and that the five-minute single-conversation prompt cache is structurally unable to reach across sub-agent boundaries. The implication is that M2's largest reductions must come from changing what the parent pre-pastes versus what each fresh sub-agent re-fetches — re-read elimination, not cache reliance, is the lever.

- **F1.3 is split into F1.3a (per-case totals + baselines) and F1.3b (per-sub-agent attribution) because the stream-json contract is not yet observed.** SD-001 records the risk that the Claude CLI attributes sub-agent token usage only to the parent transcript rather than per-dispatch. F1.3a delivers value regardless of how SD-001 resolves and is the M1 completion gate; F1.3b is contingent on a captured `evals/captures/*.events.jsonl` confirming per-dispatch usage records exist. The implication is that the rest of the program — quality nets, downgrade validation, M2 forge re-read measurement — must remain executable if F1.3b is descoped to a post-M1 follow-up.

- **Measurement infrastructure (F1.5 forge eval, F1.6 JVM fixture) sits in M1, not M2, even though the PRD originally categorized them with the architectural reductions.** The Coherence lens correction is load-bearing: the forge eval and the JVM fixture do not reduce cost on their own — they are the harness against which M2's reductions are measured. Promoting them into M1 means every M2 cost-reduction slice has a real, calibrated baseline to land against, rather than shipping on intuition and back-testing later. The implication is that M1 is a heavier milestone than the PRD initially implied, and M2 is correspondingly shorter and entirely focused on cost deltas.

- **A normative touched-files single-owner matrix is what lets M2 and M3 construct in parallel without serializing on merge conflicts.** Each prompt file, each sub-agent `model:` line, and each shared snippet has exactly one owning milestone for the program duration. M2 owns `smithy.implement.prompt` and the `smithy.forge.prompt` orchestration; M3 owns the four sub-agent model lines, the `smithy.fix.prompt` CI-log block, the new `snippets/build-output-protocol.md`, and the CONTRIBUTING.md token-delta block. The matrix is the load-bearing artifact of the Cross-Cutting Governance section below — referenced as a hard constraint by every downstream feature spec.

- **The cross-RFC dependency on expand-evals US1–US3 is a hard gate on a single slice, not on the program.** M1 measurement work and M3 template drafting both proceed independently of expand-evals; only the F3.2 model-downgrade slice is blocked until US1 (audit), US2 (mark), and US3 (cut) merge to master, because those scenarios are the quality net that validates Opus-to-Sonnet downgrades for `implementation-review`, `refine`, `plan-review`, and `clarify`. The implication if expand-evals slips materially: F3.2 either waits or rescopes its quality net to whatever planning-command scenarios are available, but never lands without at least two landing-soon scenarios per downgraded sub-agent (per SD-003). The rest of the program continues regardless.

- **The build-output protocol is `smithy.helper-docker` generalized.** The Docker skill already demonstrates the right pattern — bounded tails, captured logs, terse flags, never re-paste raw verbose output between agents — and it is the only place in the template tree where that discipline is codified. F3.1 generalizes that same shape across `npm test`, `gradle`, `pytest`, and `gh run view`, then injects the new snippet into every command that runs build, test, or CI tooling. The implication is that this is a low-risk M3 slice architecturally (the pattern is proven), but high-leverage (CI logs alone routinely run 50K–500K tokens of mostly-noise, currently unbounded).

- **The quality regression criterion is a non-negotiable gate on every M2 and M3 slice, not a stretch goal.** No slice lands that regresses structural-eval pass rate on the committed scenario set, and every sub-agent model downgrade or orchestration change requires at least two human-reviewed sampled outputs per affected sub-agent showing no quality regression before the template edit merges. The implication is that M1's measurement substrate (per-case totals + baselines from F1.3a, plus the forge and fix scenarios from F1.5 and F1.4) is doing double duty: it is both the cost signal that motivates the reductions and the quality signal that gates them.

- **No per-feature or per-bugfix PR regenerates `.claude/`; the snapshot is refreshed once per milestone by a dedicated chore.** The committed `.claude/` snapshot is allowed to drift between releases by design, per CLAUDE.md's "Source vs. Deployed Artifacts" section. F3.5 is the explicit snapshot-refresh chore at the end of M3; equivalent chore PRs land at the close of M1 and M2. The implication is that no feature or bugfix PR in this program regenerates `.claude/` or `.smithy/smithy-manifest.json` as a side effect — that coupling would force every prompt diff to drag a derived-artifact diff that obscures the actual change under review.

## Decisions

- **Preserve the PRD's three-milestone shape with M2 ∥ M3 parallel construction.** No lens identified a contradiction within the three-milestone structure itself; the parallel-construction property is a meaningful user-facing acceleration that the program is built around. Collapsing to two milestones was considered and rejected: it doubles time-to-first-savings and abandons the "low-risk quick wins decoupled from forge-eval risk" framing the PRD authored.
- **Move forge eval (F1.5) and the JVM multi-language fixture (F1.6) into M1, not M2.** The PRD originally categorized both with the architectural reductions, but the Coherence lens surfaced a PRD-internal contradiction: their justification ("needed before any forge-targeted savings can be measured" and "needed before build-output-protocol claims are anything but speculative") is the literal definition of measurement infrastructure. This is correction, not rescoping — the work content is unchanged; only the milestone boundary moves so M1 becomes the complete measurement substrate and M2 stays purely about cost deltas.
- **Split F1.3 into F1.3a (per-case totals + baselines) and F1.3b (per-sub-agent attribution).** F1.3a is low-risk and unblocks roughly 80% of downstream M2/M3 work; F1.3b carries SD-001 verification risk and only gates F3.2's rigorous per-agent quality validation. Splitting collapses the critical path: M1 completes on F1.3a regardless of how SD-001 resolves, and the rest of the program remains executable if F1.3b is descoped to a post-M1 follow-up.
- **Add F1.4 (smithy.fix eval coverage) to M1.** The PRD's success signal "smithy.forge AND smithy.fix have at least one eval scenario" was promised without an owning feature; the Completeness and Coherence lenses both surfaced this gap independently. Putting F1.4 in M1 — alongside F1.5 forge eval and F1.6 JVM fixture — makes M1 the honest measurement substrate for both highest-frequency commands.
- **Add F3.4 (PR-description token-delta protocol) and F3.5 (snapshot-refresh chore).** F3.4 closes a PRD success-signal gap ("PRs show a delta against baselines in their description") that had no owning feature; F3.5 makes the program's end-state explicit and honors CLAUDE.md's "Source vs. Deployed Artifacts" policy that the committed `.claude/` snapshot is refreshed only by dedicated chore PRs.
- **Ship all four sub-agent model downgrades in F3.2** — `smithy.implementation-review`, `smithy.refine`, `smithy.plan-review`, `smithy.clarify` all move from Opus to Sonnet. The Scope Minimalism lens recommended deferring `plan-review` and `clarify` for blast-radius reasons, but SD-003's per-agent side-by-side validation gate applies equally to all four; deferring two halves the M3 cost-reduction multiplier without proportionate risk reduction. The 5× Opus-vs-Sonnet cost gap on these scan-and-triage roles is the single biggest dial in the codebase.
- **Limit the multi-language fixture (F1.6) to JVM only.** The PRD's own assumption #4 accepts JVM as sufficient to validate the gradle clauses of the build-output protocol; Cargo, Go, and Python fixtures are deferred as nice-to-have follow-up work.
- **Express F1.2 (planning-command eval coverage) as a cross-RFC dependency, not as a feature this RFC owns.** The in-flight `specs/2026-05-03-005-expand-evals-coverage-planning-and-audit/` spec owns audit/mark/cut/render/ignite/spark scenarios; absorbing them into this RFC would duplicate ownership and create cross-RFC merge conflicts. Reference-only preserves the quality net F3.2 needs without taking ownership.
- **Adopt the touched-files single-owner matrix as a normative Cross-Cutting Governance artifact, not as SD-007 commentary.** The matrix assigns each prompt file, each sub-agent `model:` line, and each shared snippet to exactly one owning milestone for the program duration, pre-empting the M2/M3 parallel-construction merge surface PRD SD-007 flagged. Coherence lens identified the SD-007 gap; this decision closes it.
- **Reject the F2.3 competing-implementations A/B (pre-paste vs per-task brief).** The Parallelism lens surfaced running both strategies concurrently with reconcile; rejected because cost-doubling on the largest forge cost driver is anti-thematic for a token-savings program. The implementation choice is made post-M1 against F1.3a + F1.5 numbers at F2.3 spec-drafting time, not at RFC time.
- **Reject pre-M1 shipping of F3.3 (smithy.fix CI-log handling).** Parallelism lens noted the change is a single-line zero-dependency edit shippable immediately; rejected because the program's contract is that every PR carries a measured token delta. Shipping un-baselined sets an anti-precedent the Cross-Cutting Governance section explicitly forbids.
- **Reject adding F2.7 (smithy.cut Phase 2.8 measurement-only feature) and an in-program `smithy.survey` model-decision feature.** Completeness lens surfaced both as in-program candidates; both are rejected. Cut Phase 2.8 fan-out is already captured as a PRD Open Question for a follow-up RFC, and M1 baselines will surface the data automatically without needing F2.7 to exist. `smithy.survey` is invoked rarely (does not warrant a slice under the `fix > forge > cut > rare planners` hierarchy) and its work shape — web-research synthesis — is meaningfully different from the four scan-and-triage agents in F3.2.
- **Adopt the calibrated-target framing for the 40% per-slice reduction goal.** Carry the number into Goals as "40% (target; calibrated post-M1 baseline per SD-008)" rather than downgrading to "TBD" — the order-of-magnitude estimate is load-bearing for prioritization, and the calibration footnote is sufficient signal that the threshold may reset once M1 baselines land.

## Open Questions

- Should `smithy.implement` eventually migrate from Opus to Sonnet, or is Opus genuinely load-bearing for TDD synthesis quality? The decision criterion (per SD-004) is whether F2.3's spec re-read reduction plus a Sonnet downgrade together regress structural quality; the data does not exist until post-F2.3. Deferred to a follow-up RFC named in this program's future-work tracking once M2 closes.
- Is `smithy.cut`'s Phase 2.8 competing-lens fan-out (three parallel slice agents plus reconcile) worth its cost relative to a single-lens variant? F2.7 was considered and rejected as in-program scope. M1 baselines for the cut scenario (delivered via the expand-evals dependency) will quantify Phase 2.8's share of cut cost; if it is materially larger than the upstream analysis estimated, a follow-up RFC scoped to that reduction becomes the next obvious step.
- Where does the Anthropic Messages Batches API integration belong in the broader Smithy cost roadmap? Out of scope for this RFC because Batches reduces per-message price, not per-message token count — orthogonal to every lever this program pulls. Captured here so the decision is not lost between RFCs; reopened as its own RFC if and when per-message-price reductions become the next dominant cost dial.
- Should the build-output protocol expand to cover Cargo, Go, and Python build tools beyond the F1.6 JVM addition? Deferred until at least one of those toolchains becomes a sustained user demand; PRD assumption #4 accepts JVM-only as sufficient for this program.

## Cross-Cutting Governance

This section codifies program-wide constraints that apply across every milestone. Each downstream feature spec produced from this RFC inherits these constraints by reference.

### Relationship to Other Specifications

This RFC has a hard read-only dependency on `specs/2026-05-03-005-expand-evals-coverage-planning-and-audit/`. The expand-evals spec owns audit, mark, cut, render, ignite, and spark eval scenarios, the YAML scenario loader, and the baseline library. This RFC owns the token-usage reporting addition to `EvalReport`, the `smithy.forge` and `smithy.fix` eval scenarios, the JVM multi-language fixture, and the cross-cutting build-output protocol. No slice in this RFC edits files under the expand-evals spec folder; references are by path only.

M3 F3.2 (sub-agent model downgrades) takes a hard prerequisite on expand-evals US1 (audit-flawed-spec), US2 (mark-from-features), and US3 (cut-from-spec) being merged to master before any model-line edit lands in `src/templates/`. M1, M2, and the remaining M3 slices have no expand-evals dependency.

### Operational Constraints

- No M1, M2, or M3 slice regenerates `.claude/` or `.smithy/smithy-manifest.json` as a side effect of its feature or bugfix PR. Source-of-truth template edits land under `src/templates/`; eval-framework changes land under `evals/`; the F3.4 PR-description protocol lands under `CONTRIBUTING.md` — but the committed `.claude/` snapshot is never regenerated except by the dedicated chore PRs described below.
- F3.5 is the dedicated `.claude/` snapshot-refresh mechanism. Equivalent chore PRs land at the close of M1 and M2 as well — one snapshot refresh per milestone, never per feature.
- Reference: CLAUDE.md "Source vs. Deployed Artifacts" section.
- No slice ships un-baselined: every M2 and M3 PR includes a token-delta line in its PR description (per the F3.4 protocol) once F3.4 is merged.
- Every model downgrade and orchestration change passes the quality regression gate (no structural-eval pass-rate regression, at least two human-reviewed sampled outputs per affected sub-agent showing no quality regression) before the template edit merges.

## Milestones

### Milestone 1: Measurement Foundation

**Description**: Establish the measurement substrate the entire program runs on so every downstream cost-reduction slice lands against a calibrated baseline rather than intuition. M1 bundles per-case input/output token totals plus committed baselines in `EvalReport` / `evals/lib/report.ts` / `formatReport` (F1.3a), per-sub-agent attribution contingent on SD-001 evidence (F1.3b), end-to-end eval coverage for the two highest-frequency cost drivers `smithy.fix` (F1.4) and `smithy.forge` (F1.5, including the runner `git init` work that resolves SD-002), and a JVM multi-language fixture at `evals/fixture/jvm/` (F1.6). F1.1 (reference baseline evals infrastructure) and F1.2 (cross-RFC dependency on the expand-evals planning-command coverage spec) are referenced but not owned.

**Success Criteria**:
- Per-case input and output token totals render in every `npm run eval` report via `formatReport`, with committed baselines for the existing strike scenario and the landing-soon planning-command scenarios consumed from the expand-evals spec.
- The `smithy.forge` eval scenario passes end-to-end on both the existing JavaScript fixture and the new JVM fixture under `evals/fixture/jvm/`, with the runner reliably `git init`-ing the temp copy ahead of every forge dispatch (SD-002 closed).
- The `smithy.fix` eval scenario passes end-to-end against a fixture issue exercising its high-cost CI-log path, closing the PRD success-signal gap for fix coverage.
- Per-sub-agent token attribution ships in `EvalReport` and `formatReport`, or SD-001 is resolved to "parent-only attribution" with the per-case-totals fallback documented and F1.3b explicitly descoped to a post-M1 follow-up.
- The committed baseline set is sufficient to compute a token delta for any prompt or sub-agent edit landing in M2 or M3 without re-running the full eval suite by hand.

### Milestone 2: Architectural Cost Reductions

**Description**: Once M1's measurement substrate is in place, deliver the architectural cost reductions that need calibrated baselines and a forge eval to ship safely. M2 bundles a reduction in per-task spec re-reads in `smithy.implement` decided A/B between pre-paste and per-task brief at spec-drafting time against F1.3a + F1.5 numbers (F2.3), inlining `smithy.maid` into the `smithy.forge.prompt` Phase 6 orchestration to remove a sub-agent dispatch (F2.4), and a trivial-slice review-skip heuristic gated on the `<50 lines / <3 files` calibration from captured forge runs per SD-006 (F2.5).

**Success Criteria**:
- F2.3 lands with a measured per-slice token delta against the F1.3a + F1.5 forge baseline on the JS fixture and the JVM fixture, with the pre-paste vs. per-task-brief choice documented in the feature spec.
- F2.4 lands with `smithy.maid` inlined into `smithy.forge.prompt` Phase 6, the eliminated sub-agent dispatch reflected as a measured token delta against the M1 forge baseline, and no structural-eval regression on the committed scenarios.
- F2.5 lands with a calibrated threshold (initial `<50 lines / <3 files` or whatever the captured-run data supports per SD-006), the slimmed inline checklist in place of the review sub-agent on trivial slices, and a measured token delta plus at least two human-reviewed sampled outputs per affected sub-agent showing no quality regression.
- Aggregate M2 reductions move the typical `smithy.forge` slice on the JavaScript fixture toward the 40% Goals target (calibrated post-M1 per SD-008), with structural-eval pass rate unchanged on every committed scenario.

### Milestone 3: Early Cost Savings

**Description**: In parallel with M2 (gated on M1 only, conflict-free via the Cross-Cutting Governance touched-files single-owner matrix), ship the low-risk early wins visible against existing strike and landing-soon planning-command scenarios. M3 bundles the new `snippets/build-output-protocol.md` injected into `smithy.implement.prompt`, the `smithy.forge.prompt` Validation block, the `smithy.fix.prompt` Verify section, and `tdd-protocol.md` (F3.1); four sub-agent Opus-to-Sonnet model downgrades for `smithy.implementation-review`, `smithy.refine`, `smithy.plan-review`, and `smithy.clarify` gated on expand-evals US1–US3 merging to master (F3.2); failure-extraction grep for `smithy.fix` CI logs in the Verify section's `gh run view` consumption block (F3.3); a PR-description token-delta protocol added to `CONTRIBUTING.md` (F3.4); and the dedicated `.claude/` snapshot-refresh chore at program end (F3.5).

**Success Criteria**:
- The new `snippets/build-output-protocol.md` is referenced from every command that runs build, test, or CI tooling (`smithy.implement.prompt`, `smithy.forge.prompt` Validation, `smithy.fix.prompt` Verify, `tdd-protocol.md`), and raw verbose `npm test` / `gradle` / `pytest` / `gh run view` output no longer lands unbounded in any agent's context window on the committed scenarios.
- All four sub-agent model downgrades (`smithy.implementation-review`, `smithy.refine`, `smithy.plan-review`, `smithy.clarify`) land in `src/templates/` with no quality regression on at least two expand-evals scenarios per downgraded agent per SD-003, only after expand-evals US1 (audit), US2 (mark), and US3 (cut) merge to master.
- `smithy.fix` CI-log handling (F3.3) lands with the failure-extraction grep step in the `smithy.fix.prompt` Verify section's CI-log consumption block (the step that ingests `gh run view` output) and a measured token delta against the M1 `smithy.fix` baseline showing CI-log token consumption materially reduced on the committed scenario.
- The `CONTRIBUTING.md` token-delta protocol section (F3.4) is added and at least one M3 PR after it lands carries the protocol's token-delta line in its description, demonstrating the contract is followed end-to-end.
- The `.claude/` snapshot is refreshed at program end via F3.5 as a dedicated chore PR (with parallel chore refreshes at the close of M1 and M2), and no M1/M2/M3 feature or bugfix PR regenerates `.claude/` or `.smithy/smithy-manifest.json` as a side effect.
