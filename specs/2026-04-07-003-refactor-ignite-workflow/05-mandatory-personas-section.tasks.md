## Slice 1: Enforce Personas Section End-to-End

**Goal**: Harden the piecewise RFC pipeline so that personas identified during Phase 2 clarification always reach the final RFC's `## Personas` section. Tighten sub-phase 3b's smithy-prose dispatch so the clarification-sourced personas are mandatory, augment sub-phase 3g's harmonize pass to explicitly verify and repair a missing or empty Personas section, and lock both enforcement points with template tests.

**Justification**: The plumbing to produce a Personas section exists (Story 1 template slot, Story 2 smithy-prose guidance, Story 4 sub-phase 3b wiring), but nothing currently mandates substantive output or detects a silent drop between clarification and the final file. Splitting enforcement (3b) from verification (3g) would leave acceptance scenario US5-1 unverifiable mid-slice — an empty Personas section could ship unchecked. Bundling the enforcement, verification, and tests into one PR-sized slice delivers the end-to-end guarantee in a single merge.

**Addresses**: FR-005, FR-007b; Acceptance Scenarios US5-1, US5-2, US5-3

### Tasks

- [ ] **Tighten sub-phase 3b dispatch to enforce persona forwarding**

  In `src/templates/agent-skills/commands/smithy.ignite.prompt`, update the Sub-phase 3b block inside `{{#ifAgent}}` so the orchestrator explicitly requires personas surfaced during Phase 2 clarification to appear in the drafted section. Satisfies AS US5-3.

  _Acceptance criteria:_
  - 3b dispatch continues to pass `clarify_output` and `rfc_file_path` to smithy-prose
  - 3b dispatch adds a `tone_directives` value stating that named personas from clarification are mandatory
  - 3b instructs the orchestrator to halt with a diagnostic pointing at the clarification record if smithy-prose returns empty or placeholder content for Personas
  - Non-agent (`{{else}}`) path and other sub-phases remain unchanged

- [ ] **Add explicit Personas verification and repair to sub-phase 3g harmonize**

  In the same file, extend the Sub-phase 3g harmonize block so the coherence pass names `## Personas` as a mandatory section and repairs it if missing or empty. Satisfies AS US5-1 and US5-2 at the harmonize boundary.

  _Acceptance criteria:_
  - 3g instructions explicitly name `## Personas` as a verified mandatory section alongside the existing generic presence check
  - 3g instructions require Personas to be positioned after Out of Scope and before Proposal
  - If Personas is missing or empty after harmonize, 3g re-dispatches smithy-prose with the accumulating RFC and the clarification output and re-appends before finalizing
  - Generic present/non-empty verification for unaffected sections is preserved
  - Non-agent path is unchanged

- [ ] **Lock Personas enforcement with ignite template tests**

  Extend `src/templates.test.ts` to verify that the ignite claude-variant rendered output encodes both the 3b dispatch enforcement and the 3g harmonize verification for Personas. Covers AS US5-1, US5-2, US5-3 at the template-composition level.

  _Acceptance criteria:_
  - Adds assertions to the existing `'ignite with claude variant renders competing plan dispatch'` test (or a dedicated sibling test in the same describe block) rather than a standalone test file
  - Asserts the 3b block references `tone_directives` for mandatory personas
  - Asserts the 3g block names `## Personas` as a verified section and references repair via smithy-prose
  - Asserts the default (non-agent) ignite variant still omits agent-path content — existing assertions continue to pass
  - All existing template tests continue to pass unchanged

**PR Outcome**: Merging this PR guarantees that every RFC generated by the agent path of `smithy.ignite` contains a substantive `## Personas` section grounded in the Phase 2 clarification record. Sub-phase 3b halts loudly if smithy-prose returns empty personas, and sub-phase 3g catches and repairs any silent drop before the RFC is presented for review. Template tests lock both enforcement points so regressions are caught in CI.
