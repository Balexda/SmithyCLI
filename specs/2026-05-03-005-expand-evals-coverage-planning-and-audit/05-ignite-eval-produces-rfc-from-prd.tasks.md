## Slice 1: Plant the ignite-eval PRD fixture and document it

**Goal**: A scenario-isolated directory `evals/fixture/prds/ignite-eval/` exists containing a representative PRD plant (`ignite-eval.prd.md`) conforming to the canonical spark PRD template (sections `## Problem Statement`, `## Proposed Solution`, `## Target Users`, `## Success Signals`, `## Alternatives / Build-vs-Buy`, `## Assumptions`, `## Specification Debt`, `## Open Questions` — **no** `## Dependency Order`), and `evals/fixture/README.md` carries a `## Planted Parent Artifacts` row referencing the new plant directory — creating the section if it does not yet exist.

**Justification**: The PRD plant is pure static content. Slice 2's YAML references it by exact path in its `prompt` field — landing the plant first settles the slug, lets reviewers see the plant content in isolation, and ensures Slice 2's `prompt` resolves on its first authored draft. Co-locating the README documentation with the plant keeps the "this is a deliberate fixture, do not delete" signal next to the file it protects, matching the US3/US4 precedent. The slice has no code dependency on US2's runner git-init — the plant itself is pure static content and merge-safe regardless of sibling ordering.

**Addresses**: FR-004, FR-005, FR-013, FR-014; AS 5.1 (planted-fixture precondition: PRD contains the standard PRD sections that ignite's Phase 1 PRD-read step expects); FR-012 (existing fixture untouched).

## Slice 2: Author the ignite-from-prd YAML scenario

**Goal**: `evals/cases/ignite-from-prd.yaml` exists, is auto-discovered by `loadScenarios`, and `npm run eval -- --case ignite-from-prd` reports PASS against the Slice 1 plant once US2 Slice 1's runner git-init is present, with sub-agent evidence covering smithy-prose, smithy-plan, smithy-clarify, and smithy-plan-review per the spec's Sub-Agent Evidence Matrix.

**Justification**: The YAML scenario is the user-facing deliverable that closes AS 5.1, 5.2, and 5.3. It depends on Slice 1 because its `prompt` field references the Slice 1 plant by exact path. Slice 2 also has a **hard cross-story dependency** on US2 Slice 1 (runner git-init): ignite's Phase 4 issues `git commit` inside the temp fixture copy (per scout report, ignite does NOT issue `git checkout -b` — it commits to the operator-handed-off branch as-is), which fails before any one-shot output is captured if the temp copy is not a git repository with a HEAD commit. Slice 2 MUST NOT merge until US2 Slice 1 has merged — see the Cross-Story Dependencies table. AS 5.1 references a `**RFC path**:` "or equivalent terminal marker"; the scout report confirmed the actual emitted literal in RFC-only mode is `**Spec folder**:` from `src/templates/agent-skills/snippets/one-shot-output.md` (ignite's Phase 4 explicitly relabels — "use the RFC folder as the spec folder"), so the YAML anchors to `\*\*Spec folder\*\*` with an inline comment pinning the source of truth so future template drift surfaces the regression. The matrix-faithful four-entry `sub_agent_evidence` (smithy-prose, smithy-plan, smithy-clarify, smithy-plan-review) is matrix-faithful even though ignite's runtime fan-out is heavier (3 competing plan lenses + reconcile in Phase 1.5, plus prose+plan in Phase 3a–3f); the matrix is a `[Critical Assumption]` in the spec and unilateral extension at task-execution time would be artifact drift. Empirical timeout calibration, the PR success/failure regex alternation, and the `forbidden_patterns` set are YAML-field-level concerns that belong here.

**Addresses**: FR-001, FR-002, FR-003, FR-005, FR-009, FR-010, FR-011, FR-012, FR-014; AS 5.1 (one-shot-snippet headings + `**Spec folder**:` marker), AS 5.2 (sub-agent evidence for smithy-prose, smithy-plan, smithy-clarify, smithy-plan-review per spec matrix), AS 5.3 (explicit `timeout:` field — the scenario's PASS is observable only with the field present), source-fixture-checksum invariant via the runner's existing `hashDirectory` guards; SC-001, SC-002, SC-005, SC-006.
